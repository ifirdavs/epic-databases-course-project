```bash

```
What it does:

- reads users and products from the source CSV files
- generates 100 deterministic purchases using seed `33`
- randomly chooses:
  - a user
  - a product that still has stock
  - a quantity from `1–3`
- caps quantity if the product has less remaining stock
- checks that generated purchases do not make stock negative

- inserts 100 rows into PostgreSQL `orders`
- inserts 100 rows into PostgreSQL `order_items`
- decrements product stock in PostgreSQL **transactionally**
- creates 100 Neo4j `PURCHASED` relationships

If you run it again, PostgreSQL detects the same generated batch already exists and skips duplicate inserts/stock decrement.

`purchase_generator.py` is the Phase 2 script. In short, it generates fake purchases and loads them into PostgreSQL + Neo4j.

Main flow:

1. `PurchaseGenerator.__init__()`

Loads source CSV data through `DataParser`:

```python
self.users = self.parser.parse_users()
self.products = self.parser.parse_products()
```

So it knows available users, products, prices, stock, and join dates.

2. `generate_purchases()`

Creates 100 deterministic purchases by default:

```python
DEFAULT_SEED = 33
DEFAULT_PURCHASES = 100
```

For each purchase it:

- picks a random user
- picks a random product that still has stock
- picks random quantity `1–3`
- caps quantity if stock is lower
- creates stable UUIDs for `order_id` and `order_item_id`
- creates an order date after the user’s join date
- calculates `order_total`

It returns a pandas `DataFrame`, not a CSV.

3. `_validate_purchases()`

Checks generated data is valid:

- required columns exist
- purchases are not empty
- order IDs are unique
- item IDs are unique
- quantities are between `1` and `3`

4. `load_postgres()`

Loads purchases into PostgreSQL:

- Loops through purchases one by one.
- For each order:
  - opens its own `db.get_cursor()` transaction
  - checks whether that order/item already exists
  - skips it if it already exists completely
  - locks the product row with `FOR UPDATE`
  - checks stock
  - inserts into `orders`
  - inserts into `order_items`
  - decrements product stock
  - commits that one order

So if order #45 fails, orders #1–44 stay committed, and order #45 rolls back by itself.

All of this happens in one transaction through `db.get_cursor()`.

5. `sync_neo4j()`

Creates Neo4j relationships:

```cypher
(User)-[:PURCHASED]->(Product)
```

Each relationship stores:

- `order_item_id`
- `order_id`
- `quantity`
- `date`

6. `load_all()`

Runs the full Phase 2 flow:

```python
purchases = self.generate_purchases()
self.load_postgres(purchases)
self.sync_neo4j(purchases)
```

So when you run:

```bash
uv run python -m src.utils.purchase_generator
```

it generates 100 purchases, loads them into PostgreSQL, syncs them into Neo4j, and prints:

```text
Generated 100 purchases
```

# 1. Product Search with Caching

## Full-text search in PostgreSQL

Implemented items:

- Full-text search in PostgreSQL
- Cache results in Redis with 1-hour TTL
- Search by product name, description, and tags
- Filter by category and price range

Important code places:

- `src/services/search_service.py`
  - `ProductSearchService.search_products(...)`
  - Builds a normalized Redis cache key.
  - Checks Redis first.
  - Falls back to PostgreSQL.
  - Writes results back to Redis.

- `src/db/postgres_client.py`
  - Adds `products.search_vector`.
  - Adds `products_full_text_search_idx` GIN index.

- `src/loaders/relational_loader.py`
  - Populates `search_vector` during product insert/upsert.

Design explanation for demo:

- PostgreSQL is the source of truth.
- Redis is only a performance cache. If Redis fails, search still works.
- `search_vector` stores searchable tokens from `name`, `description`, and `tags`.
- GIN index makes full-text lookup efficient.
- Results are cached for `CACHE_TTL = 3600` seconds.

# 2. Shopping Cart Management

Implemented items:

- Session-based carts in Redis
- Add/remove/update items
- 24-hour cart expiration is also covered because every cart key receives `CART_TTL = 86400`

Important code places:

- `src/db/redis_client.py`
  - `add_to_cart(user_id, product_id, quantity)`
  - `update_cart_item(user_id, product_id, quantity)`
  - `remove_from_cart(user_id, product_id)`
  - `get_cart(user_id)`
  - `clear_cart(user_id)`

Design explanation for demo:

- Cart key format: `cart:{user_id}`
- Redis data type: Hash
- Hash fields are product IDs, e.g. `P001`
- Hash values are quantities, e.g. `3`
- `add_to_cart` increments quantity
- `update_cart_item` sets exact quantity
- setting quantity to `0` removes the item
- `remove_from_cart` deletes one product from the cart
- each mutation refreshes the TTL to 24 hours

Before demo, make sure Redis is running:

```bash
docker compose up -d redis
```

Run the cart integration test:

```bash
.venv/bin/python -m pytest tests/test_phase3_cart.py -q -rs
```

What this test proves:

- cart is stored as a Redis Hash
- adding an existing product increments quantity
- updating sets exact quantity
- quantity `0` removes an item
- removing the last item deletes/empties the cart
- negative quantities are rejected
- cart key has a TTL

# 3. Recommendation System

Implemented items:

- "Also bought" recommendations
- Personalized user recommendations
- Similar product suggestions

Important code places:

- `src/services/recommendation_service.py`
  - `also_bought(product_id, limit=5)`
  - `personalized_for_user(user_id, limit=5)`
  - `similar_products(product_id, limit=5)`

- `src/db/neo4j_client.py`
  - Neo4j Cypher queries for `also bought` and personalized recommendations.

- PostgreSQL `product_embeddings`
  - pgvector similarity query for similar product suggestions.


Design explanation for demo:

- Graph recommendations use Phase 2 `PURCHASED` relationships:
  - `(User)-[:PURCHASED]->(Product)`
- "Also bought" means:
  - product -> buyers -> other products those buyers purchased
- Personalized recommendations means:
  - target user -> purchased products -> similar users -> products not yet bought by target user
- Similar product suggestions use vector similarity:
  - product embedding -> nearest product embeddings by cosine distance

What this test proves:

- Neo4j can return "also bought" recommendations.
- Neo4j can return personalized recommendations.
- personalized recommendations exclude products already bought by the target user.
- pgvector returns similar products using embeddings.
- source product is excluded from similar-product results.

Demo 3: similar product suggestions using pgvector

```python
from pprint import pprint
from src.services.recommendation_service import recommendation_service

pprint(recommendation_service.similar_products("P001", limit=5))    # P013: Wooden Cutting Board
```

Expected thing to point out:

- Input is a product ID.
- It compares `P001`'s embedding to other product embeddings.
- Results include a `similarity` score.
- The source product `P001` is excluded.

Underlying SQL idea:

```sql
SELECT p.*,
       1 - (other.description_embedding <=> target.description_embedding) AS similarity     -- cosine similarity = 1 - cosine distance;    <=> is the pgvector operator for cosine distance
FROM product_embeddings AS target
JOIN product_embeddings AS other ON other.product_id <> target.product_id
JOIN products AS p ON p.id = other.product_id
WHERE target.product_id = 'P001'
ORDER BY other.description_embedding <=> target.description_embedding    -- ascending order of cosine distance = descending order of similarity
LIMIT 5;
```

# 4. Semantic Search

Implemented item:

- Natural language product search

Already covered by earlier work:

- Generate product embeddings: Phase 1 `src.loaders.vector_loader`
- Vector similarity search: `RecommendationService.similar_products(...)`
- "More like this" functionality: `RecommendationService.similar_products(product_id)`

Important code place:

- `src/services/semantic_search_service.py`
  - `natural_language_search(query, limit=10)`
  - encodes the user query with the same MiniLM-style 384-dimensional model
  - compares query embedding against `product_embeddings.description_embedding`

  - caches normalized query results in Redis for 1 hour
